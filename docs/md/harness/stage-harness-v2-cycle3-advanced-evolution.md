# 第三周期 (Cycle 3)：高阶演进与全面收紧

**定位**：强卡点默认化、沉淀复用；AI 相关仅作外围探索
**原则**：从“标杆已跑通 strict”走向“全局默认严苛与可复用插件”；**核心运行时**（状态机、门禁、证据归档）与 **AI 生成 Adapter** 等工具链探索解耦，避免混在同一交付面。

## 核心目标
在第二周期已证明标杆项目 **`strict` 可用**的前提下，将 **`strict` 提升为全局/工作区默认**，并关闭长期依赖 `compat` 的路径；沉淀通用的能力插件池 (Capability Packs)。**AI 自动生成 Adapter 不作为本仓库核心运行时交付**，仅列为外围探索或长期研究项（见下文）。

## 开发任务清单

### 1. 门禁全面切 Strict 模式 (Strict as Default)
*目标：在 Cycle 2 已完成标杆级 strict 验证的前提下，将强门禁从“标杆要求”升级为**默认路径**，消灭对 compat 的长期依赖。*
- [ ] 在全局或工作区级别默认启用 `strict` 模式，并将 `compat` 降级为显式开启的过渡/救急选项（或按计划移除），**本周期不承担“第一次证明 strict 能跑真实项目”**——该证明已在第二周期完成。
- [ ] 强化第七道门禁（`VERIFY`）：如果 `evidence_refs` 为空、无法解析或未关联真实可复核的证据，直接阻断流程并抛出 Error，严禁退化为仅凭 `PASS` 字段放行。
- [ ] 确保框架层的门禁只做“证据存在性、格式、索引与互证检查”，不干涉具体的业务真相逻辑。

### 2. 共性能力沉淀 (Capability Packs)
*目标：提取共性插件，方便未来新项目像搭积木一样使用。*
- [ ] 将 `oms` 中的 E2E 测试驱动逻辑提取为可插拔的 `generic-e2e-verifier` 插件。
- [ ] 将 `sqlshift` 的远端日志采集与准确率专项提取为独立的可复用 `scenario-accuracy-verifier`。
- [ ] 整理代码级别的执行修复器为通用的 `generic-code-fix-provider`。
- [ ] 在 `stage-harness` 生态中建立插件池的注册与管理机制。

### 3. AI 自动生成 Adapter（外围探索 / 长期研究，非核心交付）
*说明：**不属于 `stage-harness` 核心运行时**；不得与核心状态机、门禁、证据链逻辑混编或作为本周期 DoD。可作为独立工具仓、实验脚本或文档中的远期议题推进。*
- [ ] （可选）构建专门用于“接入新项目”的子级 Agent 或外部工具链（如 `adapter-generator`），与框架发布解耦。
- [ ] （可选）扫描代码库、半自动填补 `.harness` 清单或生成薄层 Adapter 草稿；**必须经过人工审计**后再进入生产仓库。
- [ ] 若开展：在路线图/发布说明中明确其与 `stage-harness` 核心的版本与边界，避免用户误以为“装 harness 即带 AI 生成”。

## 验收标准 (Definition of Done)
1. **严格默认**：新工作区或新项目默认落在 `strict` 路径下；伪造证据或跳过测试的产物在门禁处被立即打回（与 Cycle 2 标杆验证衔接，而非本周期首次试跑 strict）。
2. **能力复用**：一个新的普通 Web 后端项目接入，无需重新手写大量 Adapter，直接组装官方沉淀的 `repo-execution` + `test-suite-verifier` + `generic-code-fix` 即可顺畅运转。
3. **（非必选）AI 辅助接入**：若单独交付工具，仅要求文档与边界清晰；**不作为第三周期核心验收项**。
