# stage-harness 对 OMS / SQLShift 覆盖能力总结

本文用于总结一个核心问题：

- 按 `stage-harness` **目前的实现**
- 它是否能够**完整覆盖** `ai-programmer-oms` 和 `ai-programmer-sqlshift` 的整个开发流程
- 并达到它们当前已经具备的**实际落地效果**

## 一句话结论

结论是：

- **不能完全支持**
- **可以覆盖相当大一部分流程骨架**
- 但如果目标是达到 `oms` / `sqlshift` 当前那种真实项目里的落地效果，`stage-harness` 目前还做不到

更准确地说：

- `stage-harness` 更适合作为**上层流程框架**
- `oms` 和 `sqlshift` 更像已经做深做实的**项目专属执行系统**

## 一、为什么不能直接完全覆盖

问题的关键不在于 `stage-harness` 有没有：

- `CLARIFY`
- `SPEC`
- `PLAN`
- `EXECUTE`
- `VERIFY`
- `FIX`

而在于：

- `stage-harness` 当前更偏**流程治理框架**
- `oms` / `sqlshift` 当前更偏**贴近具体仓库、具体工具、具体部署环境的落地流水线**

也就是说，三者的抽象层级并不一样。

## 二、总判断表

| 项目 | 能否被 `stage-harness` 覆盖流程骨架 | 能否直接达到当前落地效果 | 我的判断 |
|---|---|---|---|
| `ai-programmer-oms` | 大体可以 | 不能直接达到 | 能覆盖“流程结构”，但达不到 `oms` 现有脚本化落地深度 |
| `ai-programmer-sqlshift` | 部分可以 | 明显不能直接达到 | 流程结构可映射，但产品专属能力缺口更大 |

## 三、抽象层级对比

| 维度 | `stage-harness` | `ai-programmer-oms` | `ai-programmer-sqlshift` | 我的判断 |
|---|---|---|---|---|
| 本质 | 通用流程框架 | 通用产品研发流水线 | SQL 转换产品专用流水线 | `stage-harness` 更抽象，后两者更贴工程落地 |
| 强项 | 阶段治理、状态推进、问题分类、验收闭环 | 文档收敛、技术方案落地、构建热更、E2E 修复 | 专用部署、专项验证、前后端归因、准确率闭环 | 三者的强项不在同一层 |
| 抽象层级 | 高 | 中 | 更低、更贴业务 | `sqlshift` 最接近真实运行环境 |
| 默认能力 | 通用 | 已经有专用脚本和工具链 | 已经有更强专用脚本和工具链 | `stage-harness` 还缺“项目专属能力层” |

## 四、对 OMS 的覆盖判断

### 总体判断

`stage-harness` 对 `oms` 的覆盖情况可以概括为：

- **前半段流程治理可以覆盖**
- **后半段工程落地不能直接替代**

### 具体对比表

| 能力 | `stage-harness` 当前情况 | 是否足够替代 `oms` | 我的判断 |
|---|---|---|---|
| 需求澄清 | 有，且治理更强 | 可以覆盖 | `stage-harness` 在前段并不弱 |
| 需求/spec 收敛 | 有 | 可以覆盖 | 可以承接 `oms` 的需求分析骨架 |
| 方案规划 | 有 `PLAN` | 基本可以覆盖 | 能覆盖规划层逻辑 |
| 执行阶段 | 有 `EXECUTE` | 只能覆盖骨架 | 不能直接替代 `oms` 的仓库级执行脚本 |
| 构建/热更 | 没有 `oms build-deploy` 这套现成编排 | 不能直接替代 | 缺工程落地层 |
| E2E 用例生成 | 没有现成 `case list` 生成器 | 不能直接替代 | 缺验证生成器 |
| E2E 执行 | 没有 `oms_e2e_tools` 这类专用工具链 | 不能直接替代 | 缺测试执行器 |
| 自动修复 | 有 `FIX` 思路 | 不能直接替代 | 缺 `oms` 那套 bug 判因与生效动作 |
| 项目释放/清理 | 没有现成统一编排 | 不能直接替代 | 缺收尾工具链 |

### 对 OMS 的结论

一句话说：

- `stage-harness` 可以作为 `oms` 的**上层流程母版**
- 但不能按当前实现直接替掉 `oms` 的**工程执行层**

## 五、对 SQLShift 的覆盖判断

### 总体判断

`stage-harness` 对 `sqlshift` 的覆盖情况比 `oms` 更弱。

原因不是前半段流程完全对不上，而是 `sqlshift` 的后半段已经高度产品化：

- 专用部署
- UI/API/UI+API 分层验证
- 前后端缺陷归因
- 远端日志采集
- SQL 准确率专项闭环

这些都不是 `stage-harness` 当前原生能力。

### 具体对比表

| 能力 | `stage-harness` 当前情况 | 是否足够替代 `sqlshift` | 我的判断 |
|---|---|---|---|
| 需求澄清/计划 | 可以映射 | 部分可以 | 前半段能覆盖一部分 |
| 执行阶段 | 可映射到 `EXECUTE` | 只能覆盖骨架 | 不能直接替代仓库执行和部署细节 |
| 专用构建部署 | 没有专用 profile dispatch / deploy 后端 | 不能替代 | 缺专用部署器 |
| 真实改动驱动测试生成 | 没有现成能力 | 不能替代 | 缺真实改动分析器 |
| UI/API/API+UI 分层验证 | 没有现成能力 | 不能替代 | 缺测试分层能力 |
| 前后端 bug 归因 | 没有现成能力 | 不能替代 | 缺故障分类器 |
| 远端日志采集 | 没有现成能力 | 不能替代 | 缺远端观测链路 |
| SQL 准确率专项链路 | 没有现成能力 | 明显不能替代 | 这是最大缺口之一 |
| 验证-修复-复验闭环 | 没有现成能力 | 明显不能替代 | 缺专项闭环执行器 |

### 对 SQLShift 的结论

一句话说：

- `stage-harness` 更不可能直接替代 `sqlshift`

因为 `sqlshift` 后半段不是简单的“研发流程”，而是：

- **围绕 SQL 转换产品构建出来的专项执行与验证系统**

## 六、分阶段覆盖判断

| 阶段 | `stage-harness` 是否可覆盖 `oms` | `stage-harness` 是否可覆盖 `sqlshift` | 我的判断 |
|---|---|---|---|
| 需求澄清阶段 | 大体可以 | 部分可以 | 前半段问题不大 |
| 规格/方案收敛阶段 | 可以 | 部分可以 | `sqlshift` 自身前段就有压缩 |
| 执行阶段 | 只能覆盖骨架 | 只能覆盖骨架 | 两边都缺项目专属执行器 |
| 构建/部署阶段 | 不能直接覆盖 | 不能直接覆盖 | 缺部署编排与环境适配 |
| E2E 验证阶段 | 不能直接覆盖 | 明显不能直接覆盖 | 缺测试工具链和验证器 |
| 自动修复阶段 | 只能覆盖治理思路 | 不能覆盖专项闭环 | 缺真实修复执行链 |
| 收尾阶段 | 部分可以 | 部分可以 | 能保留框架思路，但没有现成专用工具 |

## 七、最关键的差距到底是什么

最关键的差距不是“阶段数量”，而是下面这些能力层。

| 缺失层 | `stage-harness` 当前状态 | 对 `oms/sqlshift` 的影响 |
|---|---|---|
| 项目专属执行器 | 缺 | 无法直接完成仓库级真实改动落地 |
| 项目专属构建/部署器 | 缺 | 无法复现 `build_deploy` 实际效果 |
| 项目专属 E2E 工具链 | 缺 | 无法完成真实测试执行 |
| 项目专属自动修复器 | 缺 | 无法达到现有自动修复闭环深度 |
| 项目专属清理/释放工具 | 缺 | 无法形成完整收尾能力 |
| 产品专属专项验证能力 | 明显缺 | 对 `sqlshift` 尤其关键 |

## 八、最现实的定位

如果按当前实现看，`stage-harness` 最适合扮演的角色不是：

- 直接替代 `oms`
- 直接替代 `sqlshift`

而是：

- **统一上层流程模型**
- **统一阶段治理规则**
- **统一问题分类与验收框架**

然后在下面接：

- `oms` 专用执行器
- `sqlshift` 专用执行器

换句话说，最现实的架构应该是：

| 层次 | 更适合由谁承担 |
|---|---|
| 上层流程治理 | `stage-harness` |
| OMS 项目专属执行/验证 | `ai-programmer-oms` 或其适配器 |
| SQLShift 项目专属执行/验证 | `ai-programmer-sqlshift` 或其适配器 |

## 九、最终结论表

| 结论项 | 判断 |
|---|---|
| `stage-harness` 能否覆盖 `oms/sqlshift` 的阶段结构 | 能，大体可以 |
| `stage-harness` 能否按当前实现完整跑通它们的真实开发流程 | 不能 |
| `stage-harness` 能否直接达到它们现有落地效果 | 不能 |
| `stage-harness` 最适合扮演的角色 | 上层流程框架 / 母版编排器 |
| 真正缺的内容 | 项目专属 adapter、toolchain、deploy、E2E、repair、cleanup 能力层 |

## 十、我的最终判断

一句话概括就是：

- `stage-harness` **适合统一流程形态**
- 但不适合在**不扩展能力层**的前提下，直接替代 `oms/sqlshift` 的真实工程落地效果

所以如果未来目标是让 `stage-harness` 真正覆盖 `oms` 和 `sqlshift`，合理路线不是：

- 用 `stage-harness` 直接替掉它们

而是：

1. 用 `stage-harness` 统一阶段模型与治理规则
2. 为 `oms` 挂接专用 `build_deploy`、`e2e`、`auto_repair` 适配器
3. 为 `sqlshift` 挂接专用 `deploy`、`accuracy-test`、`log-analysis`、`frontend/backend repair` 适配器

换句话说：

- **不是让 `stage-harness` 替代它们**
- **而是让 `stage-harness` 编排它们**
