# stage-harness 在 Claude Code 上借助大模型的 AI 驱动运行模型

本文是对前序方案的一次重新收敛，核心目的是把“AI”这个概念落到真实运行环境中，避免继续抽象化表述。

这里所说的 AI，不是一个独立于 `stage-harness` 之外的新系统，而是：

- `stage-harness` 作为插件运行在 `Claude Code` 上
- `stage-harness` 借助大模型完成理解、生成、诊断、修复和升级判断
- 人类默认不主动介入流程，只在 `stage-harness` 主动升级时参与决策

因此，这份方案要回答的不是“要不要用 AI”，而是：

- `stage-harness` 在 Claude Code 上，哪些能力应借助大模型
- 哪些能力必须由 `stage-harness` 自己做成确定性内核
- 人应该如何从“流程参与者”变成“按需决策者”
- 新项目如何在低人工成本下接入，并持续扩展到未来项目

## 一句话结论

最合理的目标形态不是：

- 人主导流程，AI 只是辅助写代码

而是：

- `stage-harness` 作为运行在 `Claude Code` 上的通用研发编排器
- 借助大模型完成高认知、高不确定性的任务
- 用确定性的运行时、门禁、证据和协议约束模型输出
- 只有当系统识别到高风险、高权限、高不确定性场景时，才主动反馈给人做决策

一句话概括：

- **`stage-harness` 驱动流程**
- **大模型提供认知能力**
- **人只在被主动升级时决策**

## 一、问题重新定义

在这套新定义里，必须把 3 个层次明确拆开：

| 层次 | 角色 | 负责什么 |
|---|---|---|
| 宿主层 | `Claude Code` | 提供读写文件、执行命令、调用工具、管理会话的工作环境 |
| 编排层 | `stage-harness` | 负责状态机、阶段切换、运行时治理、产物契约、证据门禁、回路控制 |
| 认知层 | 大模型 | 负责理解项目、生成方案、诊断失败、生成适配层、决定是否升级给人 |

如果这三层不分清，方案很容易走偏：

| 偏差 | 后果 |
|---|---|
| 把确定性控制也交给模型 | 流程不稳定，无法审计，难以复现 |
| 把人默认放在流程里 | 无法形成真正的 AI 主导流程 |
| 把模型当成独立系统 | 忽视 `stage-harness` 才是主控内核这一事实 |

所以新的总公式应该是：

- **`Claude Code` 是宿主**
- **`stage-harness` 是主控编排内核**
- **大模型是 `stage-harness` 的认知与生成引擎**
- **人是按需调用的决策节点，不是默认参与者**

## 二、重新收敛后的目标定位

按这个前提，`stage-harness` 的最终定位不应只是“一个可配置插件”，而应同时承担下面 4 个角色：

| 角色 | 说明 |
|---|---|
| 通用流程内核 | 统一 `CLARIFY / SPEC / PLAN / EXECUTE / VERIFY / FIX / DONE` 的状态推进 |
| 通用运行时 | 管理 `run_id`、上下文、资源、证据、回收和并发隔离 |
| AI 驱动接入工厂 | 借助大模型自动理解项目、生成 profile、生成 adapter/provider、自动验证接入 |
| 决策升级器 | 当遇到高风险、高权限、高不确定性问题时，主动生成决策包反馈给人 |

这意味着：

- 前半段继续统一流程
- 后半段继续统一协议和运行时
- 项目差异不再主要由人工编码处理，而是由 `stage-harness + 大模型` 自动完成适配和修复

## 三、大模型应该负责什么

大模型最适合处理高认知、高变化、需要理解和生成的任务。对 `stage-harness` 来说，建议把下面这些能力定义为“模型负责”：

| 能力 | 说明 |
|---|---|
| 项目理解 | 扫描仓库结构、脚本、配置、文档，形成 `project-profile` 候选 |
| 需求理解 | 在 `CLARIFY / SPEC / PLAN` 中整理问题、约束、风险、方案 |
| 能力匹配 | 判断当前项目更适合复用哪些 provider，哪些地方需要薄 adapter |
| 适配生成 | 自动产出 `project-profile.yaml`、adapter skeleton、provider 组合配置 |
| 失败诊断 | 分析 build/test/e2e/deploy 失败的根因类型 |
| 修复建议 | 生成最小修复方案，并驱动 `FIX` 阶段循环 |
| 决策升级 | 当问题无法自动收敛时，生成结构化 `decision package` 反馈给人 |
| 能力沉淀建议 | 识别哪些项目专属能力已经稳定，值得抽象回通用 provider |

这些能力的共同点是：

- 需要理解上下文
- 需要做模式识别
- 需要在不完整信息下推断
- 需要生成结构化方案或文本

这正是大模型擅长的部分。

## 四、哪些能力必须由 stage-harness 自己确定化

为了让系统稳定可审计，以下能力不能主要依赖模型自由发挥，而必须由 `stage-harness` 做成确定性内核：

| 能力 | 为什么必须确定化 |
|---|---|
| 状态机与阶段切换 | 决定流程可重复性和可审计性 |
| `run_id` 隔离 | 决定并发安全和运行时可追踪性 |
| artifact contract 校验 | 决定阶段产物是否合格 |
| evidence gate | 决定能否进入下一阶段或结束流程 |
| secret / 权限 / destructive action 控制 | 决定安全边界 |
| `TEARDOWN / GC / cleanup` | 决定长期运行是否会泄漏资源 |
| `legacy / compat / strict` 模式控制 | 决定迁移是否平滑 |

一句话说清楚：

- **大模型负责理解、生成、判断**
- **`stage-harness` 负责约束、执行、校验、留痕**

也可以进一步压缩成一句更工程化的表述：

- **模型负责“想”，框架负责“管”和“证明”**

### 4.1 必须补充的硬控制前提

上面的职责划分只有在一个前提成立时才真正有效：

- `stage-harness` 不能只是“写在文档里的流程规范”
- 而必须在 `Claude Code` 运行过程中拥有**外置的、可阻断的硬控制机制**

在 Claude Code 宿主上把上述硬控制**做实**，通常是**首要实现难点**（权限边界、工具拦截、阶段卡点与审计的一致性）；但它应被当作**必须攻克的工程落地点**，**不应**被误读为「因此整套主控编排架构在宿主内不可行」的证明。

这意味着至少要具备下面几类能力：

| 硬控制点 | 作用 |
|---|---|
| 外置状态机 | 当前处于哪个阶段、是否允许切阶段，不由模型自行口头声明决定 |
| 阶段卡点 | 上一阶段产物不满足 contract 时，下一阶段不能继续 |
| 工具权限边界 | 不同阶段允许调用哪些动作，由框架控制，不由模型自由决定 |
| 失败即熔断 | 关键 gate 不通过时，必须中止或回退，而不是继续“试试看” |
| 审计留痕 | 每次阶段进入、退出、阻断、升级都要可追溯 |

如果没有这些硬控制点，那么所谓“`stage-harness` 主控、大模型辅助”就会退化成：

- 大模型仍然掌握实际控制流
- `stage-harness` 只是一个被动解释器

这不符合本文的目标定位。

## 五、前半段与后半段如何重新分工

### 5.1 前半段：统一认知流程

`CLARIFY / SPEC / PLAN` 仍然是 `stage-harness` 最能发挥通用性的部分。

在这一段中：

- `stage-harness` 负责阶段编排和产物约束
- 大模型负责理解需求、代码、上下文和历史产物
- 输出仍然沉淀为结构化结果，而不是停留在聊天结论

这一段的目标不变：

- 形成清晰需求
- 明确约束与边界
- 收敛方案
- 显式暴露风险
- 准备后续执行所需决策包

### 5.2 后半段：统一运行时与协议

`EXECUTE / VERIFY / FIX / DONE` 的重点不再是统一所有项目的具体动作，而是统一以下内容：

| 统一对象 | 说明 |
|---|---|
| 运行时 | `run_id`、上下文、资源登记、回收机制 |
| 协议 | `execution-summary.json`、`verification.json`、`fix-summary.json`、`delivery-manifest.json` 等 |
| 门禁 | 结构合法性、证据完整性、阶段互证、防空转规则 |
| 回路 | `VERIFY -> FIX -> VERIFY`，以及必要时回退 `PLAN` / `SPEC` |

而项目差异则通过：

- `project-profile`
- provider
- adapter

来承接。

这里的关键不是“让模型随便执行”，而是：

- `stage-harness` 先定义统一边界
- 大模型在这个边界内自动补齐项目差异

### 5.3 验证阶段必须避免“自己写、自己判、自己放行”

这是整套方案最容易被低估、但实际风险最高的一点；在后续落地顺序中，应视为**当前最应优先落实的红线**（优先于扩展自动化接入与更多 provider 花样）：若 VERIFY 失守，后续自动化只会放大错误置信度。

如果同一个模型上下文既负责：

- 生成代码
- 生成或挑选验证方式
- 解释验证失败
- 宣布“已经修好”

那么系统很容易落入“既当运动员又当裁判”的盲区，出现静默失败。

因此，`VERIFY` 阶段必须引入**独立于本轮生成行为的客观门禁**。至少应满足下面的约束：

| 约束 | 说明 |
|---|---|
| 优先使用客观证据 | 如项目原有测试、独立 verifier 脚本、CI 结果、已有验收脚本 |
| 不允许只靠模型主观判断过关 | “我认为没问题了”不能作为 gate 通过依据 |
| 新增测试不能成为唯一放行依据 | 尤其不能只靠本轮模型刚写出的测试来证明正确性 |
| 验证结果应可复跑 | 后续任意人或流程应能重复执行验证 |
| 高风险场景引入独立视角 | 必要时使用独立 reviewer prompt、独立 verifier 或对抗性检查 |

换句话说：

- `FIX` 可以高度依赖模型
- 但 `VERIFY` 不能只依赖同一个模型上下文的主观自证

否则 “human-on-demand” 最终会退化成“人永远不会被召回，但错误已经被放过去了”。

## 六、人的角色应该如何变化

这是本轮收敛最重要的调整之一。

旧思路容易默认：

- 人来盯流程
- 人来配项目
- 人来写 adapter
- 人来判断什么时候该继续、什么时候该停

按你的初衷，这都不对。新的默认机制应该是：

- **不是 `human-in-the-loop by default`**
- **而是 `human-on-demand`**

也就是：

- 人不是流程操作者
- 人不是项目接入实施者
- 人不是默认审查员
- 人是被 `stage-harness` 主动唤起的决策者

### 6.1 新的人机分工

| 角色 | 默认职责 |
|---|---|
| `stage-harness + 大模型` | 扫描项目、生成画像、组装 provider、生成 adapter、执行、验证、修复、输出结果 |
| 人 | 只在被升级时，确认高风险决策、权限授权、策略取舍 |

### 6.2 AI 主动反馈给人的三种级别

为了避免“什么都找人”，建议把反馈机制分成 3 个级别：

| 级别 | AI 行为 | 是否阻塞流程 |
|---|---|---|
| `auto-execute` | 直接继续执行 | 否 |
| `auto-notify` | 主动同步状态、风险、预计动作 | 否 |
| `decision-required` | 输出决策包，请人拍板 | 是 |

### 6.3 只有这些场景才应升级给人

| 场景 | 是否需要人工决策 | 原因 |
|---|---|---|
| 生产部署 | 是 | 影响真实环境 |
| 申请高权限 secret / 凭证 | 是 | 涉及安全边界 |
| destructive action | 是 | 涉及不可逆风险 |
| 需求或方案多轮无法收敛 | 是 | 需要业务判断 |
| 自动修复连续失败且成本持续升高 | 是 | 需要是否继续投入的决策 |
| 普通 build/test/e2e 失败 | 否 | 应先由 AI 自修 |
| 普通项目接入缺适配层 | 否 | 应先由 AI 自动补 adapter |

这样人不再是默认常驻在流程里的操作员，而是一个明确的决策闸门。

### 6.4 必须量化“什么时候强制召回人”

如果只写“连续失败且成本升高时召回人”，工程上很容易失控，因为：

- “连续失败”可以被模型解释成“再试一次”
- “成本升高”如果没有参数化阈值，就会导致死循环

因此建议把人工召回条件落成明确的运行策略，例如：

| 触发器 | 建议规则 |
|---|---|
| FIX 重试次数 | 同一问题分类连续重试超过 3 次，强制升级 |
| 错误模式重复 | 连续 2 次出现同类核心错误栈或同类 gate 失败，强制升级 |
| 改动膨胀 | 修复改动明显偏离最小修复原则时，强制升级 |
| 权限升级 | 一旦需要更高权限、真实外部资源或生产环境访问，强制升级 |
| 验证失真 | 缺少独立客观证据，或只有模型自证，强制升级 |

这样“human-on-demand” 才是一套可执行的策略，而不是一句方向性口号。

## 七、借助大模型后的新项目接入模型

在这套模式下，新项目接入不应该再理解为“人工做项目适配”，而应理解为：

- `stage-harness` 借助大模型自动完成项目接入

**阶段预期**：借助模型生成 `project-profile`、adapter 骨架等是**长期目标**；**第一阶段应优先通过手写官方 adapter** 跑通协议与运行时模型（含证据链与 gate），待契约稳定后再加大「自动生成 + 迭代修复」在主链路中的权重，避免首阶段被生成质量绑架。

但这里必须收窄预期：

- 它不应被理解为“零上下文接入任何复杂项目”
- 更合理的目标是：**标准化项目尽量自动接入，复杂存量项目采用半自动向导接入**

因此，下面 6 步流程更准确的表述应是：

- 自动优先
- 必要时一次性知识注入
- 注入后仍由 AI 继续完成适配与验证

建议把流程抽象成下面 6 步：

### Step 1：自动扫描项目

`stage-harness` 在 `Claude Code` 中读取：

- 仓库结构
- 脚本定义
- 构建配置
- 测试配置
- 部署脚本
- 文档和现有运行线索

然后借助大模型产出：

- 初版 `project-profile`
- 候选 provider 组合
- 明显风险点和不确定点

如果扫描阶段已经识别出明显缺口，例如：

- 依赖未提供的内网地址
- 依赖外部凭证
- 缺少数据库初始化知识
- 缺少真实部署拓扑说明

则不应假装“还能全自动继续”，而应主动进入一次性知识注入流程。

### Step 2：自动生成接入骨架

大模型基于扫描结果**可**生成：

- `.harness/project-profile.yaml`
- provider 绑定配置
- adapter skeleton
- outputs/context/evidence 的映射关系

在自动化成熟之前，上述产物也可由**手写模板 + 官方样本 adapter**完成等价目标。无论来源如何，这一步的目标不是一次生成完美实现，而是快速得到一个可验证的最小接入骨架。

对于复杂项目，这一步之前允许由 `stage-harness` 主动向人发起有限问题收集，例如：

- 依赖哪个数据库或中间件
- 哪些环境变量必须存在
- 哪些脚本是标准入口
- 哪些目录或服务不能碰

这里的人类输入是**一次性知识注入**，不是重新变成人工执行者。

### Step 3：以 compat 模式运行接入验证

`stage-harness` 用 `compat` 模式跑最小接入闭环，检查：

- `run_id` 是否正常创建
- 产物是否落在正确路径
- `EXECUTE / VERIFY / FIX / DONE` 的结构化输出是否成立
- evidence 是否可追溯
- `TEARDOWN / GC` 是否能收口

### Step 4：自动修复接入问题

如果接入验证失败，则优先由大模型自动完成：

- adapter 补全
- provider 参数修正
- 证据映射修正
- context 依赖修正

这里的目标是把“项目接入”也纳入 `VERIFY / FIX` 思维，而不是失败就退回给人。

但要明确边界：

- 如果失败根因是代码或配置映射问题，优先自动修复
- 如果失败根因是外部环境、权限、网络、组织知识缺失，则应直接升级给人，而不是陷入幻觉式重试

### Step 5：必要时才升级给人

只有出现高风险问题时，`stage-harness` 才主动升级，例如：

- 需要外部权限
- 需要生产环境资源
- 项目行为存在不可自动判断的歧义
- 自动接入多轮失败且修复代价持续上升

因此，新项目接入的默认模型不是“绝不找人”，而是：

- **默认自动**
- **必要时一次性问清关键知识**
- **问完之后继续由 AI 主导完成剩余适配**

### Step 6：沉淀共性能力

当多个项目反复出现相同模式时，系统应输出建议：

- 哪部分 adapter 已经稳定
- 哪部分可以抽象成通用 provider
- 哪些 profile 规则可以内置为更稳定的模板

这一步决定未来项目的接入成本能否持续下降。

但这里也要避免“AI 直接污染通用底座”，因此建议增加一层治理约束：

| 沉淀对象 | 建议策略 |
|---|---|
| 临时 adapter | 可先保留在项目侧，继续观察 |
| 多项目重复出现的能力 | 进入候选 provider 池 |
| 通用 provider 升级 | 必须经过独立验证或人工确认后才能进入平台基座 |

也就是说，AI 可以提出沉淀建议，但不应默认拥有“直接改写通用基座”的最终权限。

## 八、为什么这种模式能降低长期成本

如果继续走“每来一个项目就人工接一次”的路径，成本会随着项目数线性上升。

而这套模式的目标是把成本结构改成：

| 成本类型 | 承担者 | 变化趋势 |
|---|---|---|
| 框架底座成本 | `stage-harness` 团队 | 前期较高，但偏一次性 |
| 新项目接入成本 | `stage-harness + 大模型` | 随着经验积累逐步下降 |
| 人工成本 | 决策人 | 压缩到只剩高风险场景 |
| 平台进化成本 | 框架团队 + 大模型 | 通过能力沉淀逐渐降低边际成本 |

真正重要的不是“provider 越多越好”，而是：

- 旧 provider 能否被模型稳定复用
- 新 adapter 能否被模型稳定生成
- 接入结果能否被框架稳定验证
- 通用能力能否被持续反哺回框架

所以这套方案的本质重点并不是“做很多适配器”，而是：

- **让 `stage-harness` 借助大模型，稳定地产生、验证、修复和进化适配能力**

## 九、对后续实现的约束建议

如果按这个方向继续推进，后续落地时应始终坚持以下边界：

| 约束 | 说明 |
|---|---|
| 不把状态机交给模型 | 模型只能在框架边界内工作 |
| 不把安全边界交给模型 | secret、权限、破坏性动作必须由框架硬控 |
| 不让人默认驻场 | 人只能按需决策，不能退回到手工值守 |
| 不把项目名写死进核心逻辑 | 通过 profile/provider/adapter 处理差异 |
| 不只看文件存在 | 必须看 evidence 和可追溯性 |
| 不把临时 adapter 永久堆积 | 要有能力沉淀和升级路径 |

### 9.1 对 destructive action 的额外约束

仅仅在规则层写“遇到 destructive action 就升级给人”还不够，因为真实风险不在口头判断，而在执行托底。

因此应补充下面的实现约束：

| 约束 | 说明 |
|---|---|
| 优先在受限环境执行 | 能在临时工作区、受限权限环境、沙箱或 dry-run 中完成的，先不碰真实环境 |
| 真实环境动作必须显式声明 | 涉及删除、覆盖、释放、真实部署的动作必须被识别为显式高风险操作 |
| 不允许隐式夹带 | 不应让危险动作混在普通脚本中悄悄执行后再补报 |
| 权限升级必须阻断 | 一旦需要更高权限，必须先进入 `decision-required` |

这类约束的本质不是“更聪明地猜危险命令”，而是：

- 尽量把危险动作放进受限执行边界
- 再把无法受限的动作升级给人决策

## 十、最终结论

按这次重新梳理后的定义，`stage-harness` 的目标形态可以概括为：

- 它是运行在 `Claude Code` 上的通用研发编排器
- 它借助大模型完成理解、生成、诊断、修复和扩展
- 它用确定性的运行时、门禁、证据和协议约束整个流程
- 它默认不需要人主动参与
- 它只在识别到高风险、高权限、高不确定性问题时，主动反馈给人做决策

最终可以压缩成 4 句话：

1. `stage-harness` 继续做通用流程与运行时内核。
2. 大模型负责高认知、高变化、需要理解和生成的任务。
3. 新项目接入默认由 `stage-harness` 借助大模型自动完成。
4. 人不是流程参与者，而是被系统主动召唤的决策者。

这才更符合 `stage-harness` 运行在 Claude Code 上时的真实定位，也更符合“AI 主导、低人工参与、可持续扩展”的原始目标。
